# Configurations / 配置功能

> 配置功能很有用，但它解决的是“同一对象的受控变化”，不是所有差异都适合塞进一个文件里。

## 1. Scope and Goal / 范围与目标

本文主要讨论配置功能适合解决什么问题、不适合处理什么差异，以及在实际使用中怎样降低后期维护风险。

对个人或没有成熟数据管理体系的团队而言，配置功能确实能提高复用效率，但前提是边界清楚。否则，配置越多，后期越难排查。

## 3. Practice and Templates / 实操与模板

### 3.1 Suitable Cases / 适用场景

配置功能更适合处理“同一对象的系列化变化”，例如：

- 同一零件的不同长度规格
- 同一零件的不同状态，如毛坯与机加状态
- 同一装配体的不同显示状态或爆炸状态

这类变化仍然共享同一设计逻辑，因此用配置通常是合适的。

### 3.2 Unsuitable Cases / 不适用场景

以下情况通常不建议继续依赖配置硬撑：

- 结构路线差异明显的方案
- 特征树差异过大、难以共用的零件
- 实际上已经接近“新零件”的对象

如果继续强行放在同一个文件中，后期维护成本往往会高于复用收益。

### 3.3 Practical Rule / 实操原则

一个相对稳妥的判断方式是：

- 结构差异小，用配置
- 结构差异大，另建文件

如果还依赖设计表或较多特征状态控制，则更要定期复核各配置是否真正同步，避免出现“以为改了，实际没改全”的情况。

## 4. Other Key Points / 其余要点

### 4.1 Configuration Drift / 配置漂移

配置功能最常见的问题，不是当下不能用，而是随着修改积累，逐步出现“不同配置之间不再真正一致”的漂移现象。

这种问题往往不容易第一时间暴露，但一旦进入出图、采购或装配阶段，就会带来额外排查成本。

### 4.2 Relationship with Version Control / 与版本管理的关系

配置功能解决的是单个文件内部的受控变化，不应替代版本管理本身承担的职责。若对象已经进入明显的版本分叉阶段，通常应交由独立文件和版本管理处理，而不是继续堆在配置里。

## 5. Boundaries and Risks / 边界与风险

- 一个配置中改动特征，未必会自动覆盖到其他配置
- 配置越多，复核成本越高
- 配置文件不宜承担本该由版本管理承接的职责
- 若命名混乱，配置本身也会变得难以理解和复用

## 6. Summary / 小结

配置功能适合处理受控、可复用、结构差异较小的变化；一旦对象已经接近新零件或新方案，就应及时拆分，不必勉强继续留在同一个文件里。

配置不是越多越好，关键在于边界是否清楚、后期是否可维护。
